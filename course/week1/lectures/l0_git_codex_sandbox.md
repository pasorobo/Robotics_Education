# Git / Codex / Sandbox 最小語彙

## このLectureの目的

チーム全員がGit/GitHub・Sandbox・Codexの最小語彙と運用ルールを共有し、Week 1 Lab 0でSandboxを迷わず立ち上げられる状態にする。

## 1. Git / GitHub 最小語彙

Git操作の詳細は `docs/glossary.md` を参照。ここでは Week 1 で最低限必要な8語を示す。

| 用語 | 意味 (1-2行) |
|---|---|
| **branch** | コミット履歴の分岐。`main` を汚さずに作業するための隔離空間。 |
| **commit** | スナップショット。変更内容・理由・IDを記録する最小単位。 |
| **remote** | GitHub など、ネットワーク上にあるリポジトリの別名。 |
| **origin** | `git clone` 時に自動設定されるデフォルトのリモート名。 |
| **upstream** | フォーク元の本家リポジトリ。差分を取り込む参照先。 |
| **PR (Pull Request)** | ブランチをマージする前に差分をレビューに出す仕組み。 |
| **review** | PRの差分を確認し承認/変更要求するプロセス。 |
| **fork** | 他者のリポジトリを自分のアカウントにコピーする操作。 |

> 詳細・コマンド例は [`docs/glossary.md`](../../../docs/glossary.md) を参照。

## 2. なぜ Sandbox を作るか

教育計画 §1・§2.1 の方針要約:

- **各メンバーが自分の `Sandbox_<name>` リポジトリを所有する。** 本番リポジトリとは完全に分離し、実験・失敗を恐れず繰り返せる場所にする。
- **本番repoとの分離が必須。** Sandbox で試したコードをそのまま本番 PR に含めてはならない。必ず整理・レビューを経ること。
- **秘密情報の記載を禁止する。** APIキー・パスワード・個人識別情報は Sandbox にも書かない。GitHub の履歴は削除が困難なため、一度混入すると対応コストが高い。

Sandbox の目的は「本番リポジトリに影響を与えずに Codex との対話・prompt練習・スクリプト実験を行うこと」である。

## 3. Codex とは何か (Course 文脈)

ChatGPT Enterprise に含まれる **Codex** 機能は、チャット画面からコード生成・編集・検索を依頼できるエージェントである。

| 概念 | 説明 |
|---|---|
| **workspace** | Codex が参照するファイル群の論理的なまとまり。 |
| **GitHub connector** | Codex を GitHub リポジトリと接続し、コードを読み書きできるようにする設定。 |
| **local mode** | ローカル環境のファイルを直接操作するモード (codex CLI)。 |
| **cloud mode** | ChatGPT Enterprise の Web UI から操作するモード。履歴・設定はサーバ側に保持。 |

本コースでは cloud mode を基本とし、GitHub connector で各自の `Sandbox_<name>` に接続して使う。

## 4. Codex に頼む粒度

タスク分解の基本ルール:

- **1 PR = 1責務。** 「ファイル追加」「関数修正」「テスト追加」を同一 PR に混在させない。Codex への依頼も同じ単位で区切る。
- **勝手に依存を追加しない。** `requirements.txt` や `package.xml` の変更を伴う依頼は、事前にチームに共有してから行う。
- **検証コマンドが書ける範囲で依頼する。** 「完成したか確認するコマンド」を自分で言語化できない依頼はスコープが広すぎる。依頼前にそのコマンドを書き出すこと。

## 5. prompt前 5 項目 (重要)

Codex に依頼する前に以下の5項目を必ず確認する。詳細テンプレートは [`tools/codex_prompt_template.md`](../../../tools/codex_prompt_template.md) を参照。

| # | 項目 | 確認内容 |
|---|---|---|
| 1 | **目的** | この依頼で何を達成したいか (1文で言える?) |
| 2 | **対象ファイル** | 変更するファイル・関数・行範囲を特定しているか |
| 3 | **受け入れ条件** | 完了を判定するコマンド・テスト・目視確認方法があるか |
| 4 | **禁止事項** | 変えてはいけないAPI・依存・ブランチを明示したか |
| 5 | **ロールバック手段** | うまくいかなかった場合にどう戻すか決めているか |

Week 1 では prompt 5項目を**暗唱またはすぐ参照できる**ことが合格条件の一つである。

## 6. 委ねない判断

以下の判断は必ず人間が行う。Codexに委任してはならない (spec §2.6 と整合):

- **Affordance schema の設計。** どのロボット動作を許可/禁止するかの境界定義。
- **評価指標の選定。** 何をもって「良い結果」とするかの基準策定。
- **安全境界の決定。** 実機での速度上限・停止条件・非常停止ロジック。
- **実機投入可否の判断。** シミュレーション結果が実機展開の条件を満たすかの最終確認。

Codexが生成したコードがこれらの領域に触れている場合、必ず担当者がレビューし承認を取ること。

## 7. レビュー観点

PR レビュー時に確認する5点:

1. **diff summary** — 変更行数・ファイル数が依頼スコープに収まっているか。想定外のファイルが含まれていないか。
2. **動く根拠** — テスト・動作確認コマンドの結果が添付されているか。「動くはず」でなく「動いた証跡」があるか。
3. **壊れうる条件** — エッジケース・外部依存・環境差異によって失敗するシナリオを列挙しているか。
4. **採用しない提案** — Codexが提案したが採用しなかった変更がある場合、その理由をコメントに残しているか。
5. **追加修正** — レビュー指摘を受けた場合、同一PRに追加コミットで対応しているか (新PR不要)。

## 8. Week 1 における Codex の位置づけ

Week 1 での Codex 利用は **接続確認 + ルール理解 + prompt 5項目練習のみ** とする。

- GitHub connector のセットアップと疎通確認を Lab 0 で行う。
- prompt 5項目チェックリストを実際に書いて提出する練習を行う。
- **生成コードを成果物 (提出物・本番PR) に必須化しない。** コードを生成させても、採用するかどうかは学習者の判断に委ねる。
- **本格的なコード生成利用は W2 Lab 4b 以降。** ROS 2 基礎とTFの最低語彙を持った後に初めてCodexを活用する段階に入る。

この段階での目的は「Codexが何者か・何に使えて何に使えないか」を体感することである。

## 9. よくある誤解

**誤解1: 「Codex に任せれば早い」**

Codexは適切な依頼があれば強力だが、曖昧なスコープ・不十分なコンテキストの依頼では修正コストが依頼コストを上回る。特に初週は「prompt 5項目を埋める時間 > コーディング時間」になって当然であり、それ自体が学習の本体である。速さは正しいプロセスの副産物であって目的ではない。

**誤解2: 「Sandbox なら何書いてもOK」**

Sandboxは本番リポジトリへの影響を遮断するが、書いたコード・データ・コメントはGitHub上に残る。秘密情報・個人情報・不適切なコンテンツの記載は Sandbox でも禁止である。また、Sandbox で動いたコードが本番に流入しないよう、Sandbox → 本番のフローは必ずレビューを経ること。

**誤解3: 「PR は人間レビュー要らない」**

Codexが生成したPRであっても、マージ前に必ず人間のレビュアーが差分を確認する。自動テストがパスしただけではレビュー完了にならない。「動いている」と「正しい」は別の問いであり、安全境界・命名・設計判断は人間が責任を持つ。
